## Background

My career has been in **fintech** — payments, lending, wealth management, and trading infrastructure across consumer and enterprise products. I've owned products from zero to one, scaled platforms to millions of users, and navigated the part of fintech that doesn't make it to conference decks: compliance constraints, edge-case failures, and the slow work of building systems people trust with their money.

**What fintech taught me about systems:**
Financial products are unforgiving because money is stateful. A failed payment isn't just a bad UX moment — it has downstream effects: retries, reconciliation mismatches, fraud flags, customer trust. Working in this domain trains you to think in systems: what happens when this fails, what does the state look like after, what does recovery look like. That mental model is what I bring to every product I work on.

## AI PM — What I'm actually doing

AI changes the product contract. When the output is probabilistic, "works correctly" means something different. I'm building new muscle around three things:

**1. Defining quality for non-deterministic outputs**
A traditional acceptance criterion is binary. AI outputs are distributions. I'm developing evaluation frameworks — structured rubrics, human review pipelines, automated regression tests using LLM-as-judge — to define and measure "good" for AI features in a way that engineering teams can actually build against.

**2. Building with the API, not just speccing features for it**
I use the Anthropic Claude API and Vercel AI SDK directly. I've built streaming chat interfaces, tool-use pipelines, and prompt systems with caching and structured outputs. When I write a spec for an AI feature, I've already hit the edge cases — context window limits, latency tradeoffs, prompt injection risks, hallucination patterns. That changes the conversation with the team.

**3. Identifying where AI creates new product surface, not just automates old surface**
Most "AI features" are automation in disguise. The more interesting question is: what products are only possible because of LLMs? FinScope (below) is an early answer to that — a research workflow that couldn't exist without AI doing the synthesis work in real time.

## Projects — shipped, not planned

### [FinScope](https://github.com/achalarora/finscope)
**The problem:** Retail investors have access to data but not analysis. Professional-grade research — fundamental valuation, technical structure, Elliott Wave counts — takes hours and requires domain expertise most people don't have.

**What I built:** Enter any ticker. Get a structured analysis covering macro context, valuation, chart structure, and wave count — generated by Claude, grounded in Yahoo Finance data.

**What it demonstrates:** Fintech + markets depth (Elliott Wave, DCF, technical indicators), Applied AI (streaming LLM pipeline with structured financial prompts), Product sense (the insight is that synthesis is the bottleneck, not data access).

`Next.js` · `TypeScript` · `Anthropic Claude API` · `Yahoo Finance` · `TradingView Charts`

### [Expense Tracker](https://github.com/achalarora/expense-tracker)
**The problem:** I wanted a personal finance tool that worked offline, stored data locally, and synced to a format I could actually use on mobile — without a subscription.

**What I built:** A desktop app with full expense/credit tracking, monthly summaries, and one-click sync to Google Sheets — which acts as the mobile interface.

**What it demonstrates:** Product sense (solving for a real workflow gap, not building features), Execution (desktop app with OAuth integration, background threading, real daily use), Systems thinking (local-first with cloud sync — the data model handles partial syncs and duplicates gracefully).

`Python` · `CustomTkinter` · `SQLite` · `Google Sheets API`

---

### [Home Network Tracker](https://github.com/achalarora/home-network-tracker)
**The problem:** I wanted per-device, per-process visibility into home WiFi usage — and found that no consumer tool gave me the granularity I wanted.

**What I built:** A background collector agent that samples NIC stats and per-process TCP connections, stores them in SQLite, and generates a 7-day terminal report showing usage by device and by application.

**What it demonstrates:** Systems thinking (multi-device architecture with a sync server — secondary devices push samples to a primary collector), Execution (Windows EStats API via ctypes, SNMP probing, threaded collection with proper locking).

`Python` · `psutil` · `SQLite` · `SNMP` · `Windows EStats API`

## How I think — a few positions

> **Specs written before a prototype are fiction. The prototype is the spec.**

> **In fintech, the edge case isn't the exception — it's the product. The failure state is where trust is won or lost.**

> **Most AI features are automation with a language model bolted on. The interesting work is identifying where the AI makes an entirely new product possible — not just an existing workflow faster.**

> **A PM who can read a stack trace is a better partner to an engineering team. A PM who can ship a working prototype is a different kind of PM entirely.**
